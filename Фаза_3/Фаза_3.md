
# Фаза 3: Оценка проекта

## 1. Сценарии использования (Use Cases)

Для системы «Онлайн-магазин тортов + 3D-конструктор» детально проработано 12 сценариев использования, охватывающих полный цикл взаимодействия: от регистрации клиента до выгрузки задания на производство.

### UC-1. Регистрация нового пользователя
*   **Акторы:** Клиент
*   **Предусловия:** Пользователь не авторизован.
*   **Основной поток:** Ввод email, имени и пароля -> Проверка уникальности в БД -> Создание профиля -> Перенаправление в личный кабинет.

### UC-2. Авторизация (Вход)
*   **Акторы:** Клиент, Администратор
*   **Основной поток:** Ввод логина/пароля -> Проверка учетных данных -> Создание сессии -> Открытие соответствующего интерфейса (витрина или админ-панель).

### UC-3. Покупка готового торта
*   **Акторы:** Клиент
*   **Цель:** Оформление заказа на стандартную позицию.
*   **Основной поток:** Просмотр каталога -> Выбор торта -> Добавление в корзину -> Ввод данных доставки -> Сохранение заказа в БД.

### UC-4. История заказа и статусы
*   **Акторы:** Клиент
*   **Цель:** Отслеживание этапов приготовления (Принят, Готовится, Доставлен).
*   **Основной поток:** Открытие раздела «Мои заказы» -> Запрос списка из БД -> Отображение текущего статуса и состава заказа.

### UC-5. Восстановление пароля
*   **Акторы:** Клиент, Email-сервис
*   **Основной поток:** Запрос на восстановление -> Отправка кода на email -> Ввод кода -> Установка нового пароля.

### UC-6. Конструктор торта (3D-интерфейс) — КРИТИЧЕСКИЙ
*   **Акторы:** Клиент
*   **Цель:** Индивидуальное проектирование изделия.
*   **Основной поток:**
    1.  Запуск 3D-движка (Three.js).
    2.  Выбор формы (круг, квадрат), количества ярусов и размеров.
    3.  Выбор вкуса (начинка/бисквит) для каждого яруса.
    4.  Интерактивное наложение декора: выбор цвета покрытия, добавление надписи, загрузка изображения для сахарной печати.
    5.  Расчет стоимости в реальном времени.
    6.  Сохранение конфигурации в корзину.
*   **Особенности:** Высокая техническая сложность реализации визуализации.

### UC-7. Админ: вход и панель
*   **Акторы:** Администратор
*   **Цель:** Доступ к инструментам управления магазином.
*   **Основной поток:** Авторизация с правами доступа -> Загрузка дашборда со сводкой по заказам и остаткам.

### UC-8. Админ: редактирование торта
*   **Акторы:** Администратор
*   **Основной поток:** Выбор торта из списка -> Изменение описания, цены или состава -> Обновление данных в БД.

### UC-10. Админ: управление ассортиментом
*   **Акторы:** Администратор
*   **Цель:** Управление видимостью товаров.
*   **Основной поток:** Переключение статуса «В наличии / Снят с продажи» -> Мгновенное обновление витрины для пользователей.

### UC-9. Админ: добавление торта
*   **Акторы:** Администратор
*   **Основной поток:** Заполнение формы нового товара -> Загрузка изображений -> Сохранение в каталог.

### UC-11. Админ: редактирование заказа
*   **Акторы:** Администратор (Менеджер)
*   **Цель:** Корректировка деталей заказа.
*   **Основной поток:** Изменение статуса оплаты/доставки или контактных данных клиента по запросу.

### UC-12. Админ: выгрузка на производство — КРИТИЧЕСКИЙ
*   **Акторы:** Администратор
*   **Цель:** Трансформация виртуального заказа в производственное задание.
*   **Основной поток:**
    1.  Выбор заказа, созданного через 3D-конструктор.
    2.  Генерация спецификации: размеры ярусов, перечень ингредиентов, технические требования к декору.
    3.  Подготовка файлов для печати (изображения клиента).
    4.  Выдача PDF-файла для кондитерского цеха.

## 3. Ресурсы и планирование

### 3.1. Состав команды (5 человек)
Для реализации проекта необходимы специалисты следующих профилей:
1.  **Tech Lead / Backend:** Проектирование архитектуры БД и API.
2.  **Frontend Developer (3D + Three.js):** Ответственный за ключевую фичу — 3D-конструктор.
3.  **QA Engineer:** Обеспечение качества, тестирование графических и платежных модулей.
4.  **Бизнес-аналитик:** Проработка требований к интеграции с производством и логике заказов.
5.  **Fullstack Developer:** Разработка административной панели и поддержка бизнес-логики магазина.

### 3.2. Сроки выполнения
Расчет срока реализации при наличии команды из 5 человек:
*   **Ресурс в месяц:** 5 чел × 150 рабочих часов = **750 человеко-часов**.
*   **Срок:** $3016.31 / 750 \approx \mathbf{4 \text{ месяца}}$.

Данный срок включает проектирование, разработку, тестирование и стабилизацию продукта.

### 3.3. Приоритетность и MVP
На основе [анализа приоритетов](`приоритет.xlsx`) выработана стратегия запуска:

*   **Этап 1 (MVP):** Реализация UC-1, 2, 3 (каталог), UC-6 (базовый конструктор) и UC-10.
*   **Этап 2:** UC-4, 7, 8, 11 (расширенная админка и личный кабинет).
*   **Этап 3:** UC-12 (автоматизация выгрузки), UC-5 (автоматическое восстановление пароля), UC-9.

**Рекомендация:** Функционал выгрузки на производство (UC-12) можно временно заменить ручной обработкой, что позволит сократить время выхода на рынок на 3-4 недели.
